# /adversarial-review — 对抗式审查

用对抗式思维验证任何产出。核心是**主动站到对立面**——假设自己是攻击者、挑刺者、最坏情况的设计者，去击穿面前这个东西，而不是站在作者这边确认它「还行」。

## 什么是对抗式审查（本 skill 的内核）

顺向审查 = 站在作者这边检查，天然倾向**确认偏误**（倾向于证明它是对的，下意识忽略问题）。对抗式审查 = **强制立场对立**，把自己当成要搞垮它的对手，主动构造攻击向量。

**为什么需要对抗而不是顺向**：人（和 AI）在检查自己/同类的东西时，会有强烈的「确认偏误」——倾向于看到「能用」「差不多」「没问题」。真正危险的缺陷，几乎都不在「正常使用」的路径上，只在「极端输入」「恶意使用」「边界情况」里暴露。顺向审查走的是正常路径，所以看不见它们；对抗式审查专门走异常路径，所以能逼出它们。

**与第一性原理的对称关系**：配套的 `first-principles` skill 管**生成**——让方案从根本事实出发、方向正确；对抗式审查管**验证**——让产出能扛住真实世界的攻击和意外。前者保证「想得对」，后者保证「没漏掉」。合起来是一个完整的「生成 → 验证」闭环，缺一不可：
- 只用第一性原理不用对抗审查 → 方向对但健壮性差，上线即翻车
- 只用对抗审查不用第一性原理 → 能扛打但方向可能就是错的

## 输入

`$ARGUMENTS` 可以是：

- 代码：文件路径、目录、或粘贴的代码片段
- 文章/文档：文件路径、URL、或粘贴的文本
- 方案/决策：一段商业方案、一个技术选型、一个人生决策的描述
- 一个系统/功能：「审查 XX 功能的健壮性」

如果输入为空，让用户指定审查对象（文件/目录/粘贴内容）。

## 执行步骤

### 第一步：明确审查对象与破坏目标

先锁定「我要击穿什么」：

1. 识别审查对象的类型（代码 / 论证 / 方案 / 决策 / 系统）和它的**核心主张或关键路径**。
2. 明确「破坏目标」：什么东西坏了算我赢？对代码是崩溃/数据错/安全洞；对论证是逻辑断裂/事实错/反例成立；对方案是核心假设不成立；对决策是被忽略的风险致命。
3. 调动已有上下文（可选）：用 Grep 搜当前目录/笔记里已有的相关内容（如已知的坑、历史事故、约束条件），把这些作为攻击的弹药。没有可跳过。

### 第二步：构造攻击向量（核心步骤）

这是对抗式审查的关键。不要问「有没有问题」，而是**枚举「怎么搞垮它」**。从六个正交的攻击维度切入：

| 攻击维度 | 通俗说法 | 这个维度专找什么 |
| --- | --- | --- |
| 1. 极端输入 | 「喂 abnormal 数据」 | 超大/超小/空/负/未来时间/特殊字符——任何「正常不会出现」的输入 |
| 2. 边界与并发 | 「卡在缝里」「同时来」 | 刚好等于阈值、off-by-one、并发竞态、时序依赖、时区/编码 |
| 3. 失败与恢复 | 「半路断了怎么办」 | 中断、超时、重试风暴、部分失败、依赖挂掉、幂等性 |
| 4. 恶意与滥用 | 「故意搞你」 | 注入、越权、伪装、资源耗尽（OOM/死循环）、社会工程 |
| 5. 逻辑与事实 | 「前提真的成立吗」 | 论证的隐含前提、因果倒置、幸存者偏差、数据来源可信度、反例 |
| 6. 时间与演化 | 「以后会怎样」 | 规模变大后失效、依赖过时、技术债累积、未来场景没考虑 |

**不是每个对象都要用全部六个**——按对象类型选最相关的：
- 代码/系统 → 1、2、3、4 为主
- 文章/论证 → 5、6 为主
- 方案/决策 → 5、6 + 风险清单

**关键心法**：每个攻击向量要具体到一个**可执行的破坏场景**，不是泛泛的「可能有并发问题」，而是「两个请求同时到达，A 读到旧值、B 写入新值，A 的写入覆盖 B → 数据丢失」。越具体，越能逼出真问题。

### 第三步：走通攻击路径，验证真伪

对每个攻击向量，实际「走一遍」破坏过程，判断它是**真威胁**还是**伪问题**：

1. 从入口出发，沿着执行/推理路径走，看它会不会真的导致破坏目标达成。
2. 区分：
   - **真威胁**：路径走得通，确实会出问题 → 给出证据（代码行/逻辑链/数据）
   - **伪问题**：理论上像问题，但被某层防护挡住了，或概率极低 → 说明为什么不算
3. 对真威胁，评估**严重度**（致命 / 严重 / 一般）和**触发概率**（必然 / 偶发 / 罕见），优先报致命+必然的。

诚实很重要：对抗式审查的价值在于找到**真**问题，而不是堆砌一堆看似吓人其实不成立的「假阳性」。假阳性多了，反而稀释真问题的信号。宁可少而准，不要多而虚。

### 第四步：输出攻击报告与加固建议

不只要指出破绽，还要给「怎么补」。对每个确认的真威胁：

1. **破绽**：一句话说清哪里、怎么被攻破。
2. **证据**：具体的代码位置 / 逻辑节点 / 数据。
3. **后果**：破坏目标如何达成（崩溃？数据错？逻辑崩？）。
4. **加固**：最小可行的修复或加固方向。

### 第五步：诚实标注审查边界

对抗式审查不能保证「绝对没问题」，要明确说清：

- **已覆盖的攻击面** vs **未覆盖的**（如「这次没查权限模型」「没做性能压测」）。
- **置信度**：哪些是确定的真 bug，哪些是「疑似，建议进一步验证」。
- 不要为了显得严格而把不确定的说成确定的——这会误导决策。

## 输出结构

```markdown
# {对象} 的对抗式审查报告

## 破坏目标
{这次审查要击穿什么：什么坏了算找到问题}

## 攻击总结（一句话）
{整体健壮性判断：能扛 / 有缺口 / 脆弱}

## 确认的威胁（按严重度排序）
### 🔴 威胁 1：{名称}（致命 / 必然触发）
- 破绽：{哪里、怎么被攻破}
- 证据：{代码行 / 逻辑链}
- 后果：{破坏如何达成}
- 加固：{最小修复}

### 🟡 威胁 2：...
### 🟢 威胁 3：...

## 排除的伪问题（避免误导）
- {看似问题但其实不成立，以及为什么} —— 列这条是为了让用户知道它被想过、为什么放过

## 未覆盖的攻击面
{这次没查的部分，如权限/性能/特定依赖}

## 加固优先级
1. {先修什么}  2. {再修什么}
```

## 多 Agent 对抗（高级用法，重要）

对抗式审查的威力与**对抗密度**正相关。单个审查者仍受限于一个视角，会有盲区。因此对重要对象，强烈建议**多开 Agent 并发对抗**，每个 Agent 扮演不同攻击者：

- 在 Claude Code 里说「开启 Ultracode / 多 agent 对 XX 进行对抗式审查」，会并发起 N 个 Agent。
- 每个 Agent 分配不同攻击维度或不同攻击者人设（如「你是专门找并发 bug 的」「你是恶意用户」「你是竞争对手」）。
- 最后汇总去重，按严重度排序。

为什么有效：不同视角的攻击者盲区不重叠，并发能把单视角漏掉的问题补上。一个 Agent 漏的，另一个会抓到。这比单 Agent 顺查一遍严格得多。

## 风格要求

- **攻击者口吻**：用「我会从……下手」「假设我提交……就能……」，而不是「这里可能有问题」——立场要真的对立。
- **具体胜过抽象**：每个威胁落到可执行的场景和证据，杜绝「建议加强健壮性」这类废话。
- **区分严重度**：致命的和轻微的不要混在一起，否则用户不知道先修哪个。
- **诚实区分真伪**：明确哪些是真威胁、哪些是放过的问题，不堆砌假阳性充数。
- **引用已知约束**：搜到的历史事故、已知约束（如有），作为攻击弹药优先用。

## 示例（节选，供体会风格）

输入：「审查一个信源抓取模块的健壮性」

```
# 信源抓取 对抗式审查报告

## 破坏目标
让抓取出错、崩溃、或污染信息流。

## 确认的威胁
### 🔴 OOM 死循环（致命 / 必然，一旦触发）
- 破绽：worker 处理超大 HTML 时内存爆 → 被系统杀 → 自动重试 → 同一个大任务再爆 → 无限循环
- 证据：worker 无内存上限 + 无重试次数限制 + 自动重试机制
- 攻击路径：我提交一个 50MB 的 HTML，worker 必爆，重试必再爆，循环至耗尽资源
- 加固：限制单任务输入大小 + 重试退避上限 + 识别「必爆任务」标记不再重试

### 🔴 未来时间污染（严重 / 偶发）
- 破绽：信源文章发布时间因时区错误显示成「未来」→ 时间戳最新 → 排到信息流最前 → 进 PUSH/RSS/日报
- 攻击路径：一个时区 bug 让某文章 time = 明天，全信息流被这一篇污染
- 加固：发布时间 > now 的条目，钳制到 now 或丢弃

## 未覆盖
未查权限模型、未压测并发上限。
```
