# /first-principles — 第一性原理解读与分析

用第一性原理思维方式解读和分析内容或问题。核心是把事物拆解到不可再分的基本要素（根本问题），再从这些要素出发重建理解——不是记忆结论，而是建立可迁移的心智模型。

## 什么是第一性原理（本 skill 的内核）

第一性原理源自物理学，指「回归到最基本的、不能进一步被推论的真理，再从这里向上推理」。在本 skill 里它具体化为三个动作：

1. **剥离表象**：忽略术语、惯例、类比、「别人都这么做」的包装，追问「这里真正在解决什么问题」。
2. **找到根本问题**：把任何系统/概念拆成少数几个不可再分的根本问题。复杂系统 = 少数根本问题的组合。
3. **从根本重建**：从这些根本问题出发，重新理解每个组成部分为什么存在、它占了哪一格。

**为什么这有效**：人的认知天然倾向记忆「答案」而非「问题」。但答案会变（库会更新、框架会过时），根本问题却稳定（Web 应用永远要回答「界面怎么渲染」「数据怎么拿」）。抓住根本问题，新答案出现时你能自动归类，而不是重新学一遍。

## 输入

`$ARGUMENTS` 可以是：

- 一个主题或领域名（如「Web 技术栈」「区块链」「价值投资」）
- 一组名词/库/工具（如「React + Vite + TanStack Query」）
- 一个复杂或模糊的问题（如「为什么大模型会产生幻觉」）
- 一段内容：URL、文件路径、或粘贴的长文

如果输入为空，让用户指定一个文件/URL 作为分析对象，或等待用户补充主题。

## 执行步骤

### 第一步：调动已有上下文（可选）

在动手分析前，先看看手头是否已有相关材料——你自己积累的认知比通用知识更有价值，且能避免重复劳动。

1. 从输入提取 3-5 个关键词（含同义词）
2. 如果在某个项目/笔记目录下工作，用 Grep 搜索当前目录里的相关内容（README、文档、历史笔记等）
3. 读取最相关的几篇（最多 8 篇）
4. 记下：已有材料里**已有**哪些相关认知、**缺失**哪些。后续讲解时优先引用已有内容，对缺失的部分重点补充。

> 如果没有任何已有材料，直接跳到第二步——这不影响后续流程。

### 第二步：识别根本问题（核心步骤）

这是第一性原理的关键。不要罗列事实，而是追问：「这个领域/系统/概念，本质上在回答哪几个根本问题？」

方法：

- **不断追问「为什么需要它」**，直到答案变成「不这样就不行」的自明之理，那就是根本问题。
- **寻找最小完备集**：根本问题应该彼此正交（不重叠）、合在一起能覆盖整个领域。通常 3-7 个。
- **用「根本问题 = 通俗说法」的对照表述**，让非专业人士也能懂（如「界面怎么渲染」「数据怎么拿」「状态怎么记」）。

例如：
- Web 应用 → 6 个根本问题（渲染/路由/数据/状态/样式/构建）
- 一个公司 → 「创造什么价值」「卖给谁」「怎么收费」「怎么防御竞争」
- 一段论证文章 → 「核心主张是什么」「基于什么前提」「推理链条」「反例与边界」

### 第三步：从根本问题重建理解

对每个根本问题，回答：

1. **它的第一性答案是什么**：解决这个问题的最朴素思路（不绑定具体工具）。
2. **现有方案如何回答它**：具体工具/做法各自占了这个问题的哪一格，为什么这么设计。
3. **方案的分化与取舍**：同一根本问题为什么有多种答案（如 React vs Vue、Tailwind vs UnoCSS）——它们在什么维度上做了不同取舍。

这一步要建立「根本问题 → 答案」的映射表，让用户一眼看清结构。

### 第四步：建立心智模型与可迁移判断

让理解变得可迁移，而不只是懂了这一个案例：

1. **心智模型**：用一句话或一个比喻总结整个系统的运作逻辑。
2. **常用度/重要性分级**：区分「地基（必学，覆盖 80%）」「进阶（按需）」「冷门（认得即可）」——帮用户分配注意力。
3. **速读法**：给出一套可操作的「7 步看懂任何 X」模板（X = package.json / 财报 / 论文 / …），让用户下次自己能复用。
4. **关键易错点**：新人最常混淆的概念（如「服务器数据用 Query，UI 状态用状态管理库」），明确点出。

### 第五步：诚实标注不确定与边界

第一性原理强调「只把确定的东西当公理」。所以：

- 区分「事实」（可验证）和「推断」（基于证据的推测），推断要说明依据。
- 主动指出「这个领域里我不确定/有争议的部分」。
- 不为了显得深刻而强行拆解——有些东西确实是约定俗成（如语法细节），直接说明即可，不必硬套根本问题。

## 输出结构

输出一篇结构化讲解，遵循以下骨架（按输入类型可微调，但根本问题分解必须保留）：

```markdown
# {主题} 的第一性原理拆解

## 核心洞察（一句话）
{这个领域本质上在回答什么，用最朴素的话说}

## 根本问题（核心）
{N 个不可再分的基本问题，表格或列表，每个配「根本问题 = 通俗说法」}
| 根本问题 | 通俗说法 | 常用度 |
（这是全文的骨架，务必先建立这张表）

## 逐个拆解
{对每个根本问题：第一性答案 → 现有方案如何回答 → 取舍分化}

## 心智模型与可迁移判断
- 一句话心智模型
- 常用度分级（地基/进阶/冷门）
- 「N 步看懂任何 X」速读法
- 关键易错点

## 边界与不确定
{事实 vs 推断的区分；争议或我不确定的部分}
```

## 存为笔记（讲解后主动询问）

讲解完成后，主动询问用户是否存为笔记（不强制）。如果要存：

1. 生成一份结构化 markdown 笔记，含标题、日期、tags
2. tags 至少含 `first-principles` + 领域标签
3. source 记录输入来源（URL 或「以 X 为案例」）
4. 在文末「## 相关内容」列出第一步搜到的关联材料（若有）
5. 存储位置由用户决定（建议放回当前项目/笔记目录的合适位置）
6. 展示预览（标题/标签/目标位置），确认后保存

## 风格要求

- **通俗优先**：根本问题用日常语言表述，避免用术语解释术语。新人能懂是第一目标。
- **重「为什么」轻「是什么」**：解释每个东西存在的理由，而不是罗列功能。LLM 很聪明，理解了 why 能自己推导 how。
- **类比要克制**：类比是脚手架不是地基，用完要回到事物本身，避免类比失真。
- **引用已有内容**：第一步搜到的相关材料，用「你在 XX 里提到过...」引用，与通用知识区分开。
- **避免空话**：不要「这是一个非常重要的话题」这类填充，直接给洞察。

## 示例（节选自真实拆解，供体会风格）

输入：「NewsNow 的技术栈 React + TanStack Router + TanStack Query + Jotai + UnoCSS + Vite + Nitro，我刚转 Web 不熟」

输出骨架：

```
# Web 技术栈第一性原理拆解

## 核心洞察
任何 Web 应用都只在回答 6 个根本问题，所有库都是这些问题的分工答案。

## 根本问题
| 根本问题 | 通俗说法 | NewsNow 的答案 | 常用度 |
| 界面怎么渲染 | 数据→可见页面 | React | ⭐⭐⭐⭐⭐ |
| 数据怎么拿/存 | 前后端通信 | TanStack Query + ofetch | ⭐⭐⭐⭐⭐ |
| ...（共 6 行）

## 逐个拆解
### 问题 3：数据通信（分两层，新人最易混淆）
上层 TanStack Query 管「何时拿/怎么缓存」，下层 ofetch 管「怎么发请求」。
🚨 关键易错点：服务器数据用 Query，UI 状态用状态管理库，别混。

## 心智模型
库 = 分工。扫 package.json 七步法：找 react→router→query→状态库→样式库→构建工具→工程化。
```
